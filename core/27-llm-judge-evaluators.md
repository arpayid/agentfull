## 🎯 Filosofi LLM-as-a-Judge

Sebelum agen menyajikan solusi akhir, ia memanggil **instansi model penalaran terpisah (Judge)** yang bertugas murni sebagai penilai kualitas (Code Reviewer otomatis). Model Judge mengevaluasi kode berdasarkan kriteria kekokohan logika, mitigasi edge-cases, dan kepatuhan constraints tanpa dipengaruhi oleh bias pembuat kode asli. Hal ini mengurangi masuknya kode rentan (vulnerable code) ke repositori master.

---

## 🏗️ Alur Evaluasi Internal

Jika skor Judge berada di bawah batas minimal, agen secara otonom melakukan *backtracking* (Modul 08) ke tahap implementasi dengan menyertakan log review dari Judge sebagai konteks perbaikan.

```
┌─────────────────────────────────┐
│ Code Generated by Implementer   │
└────────────────┬────────────────┘
                 │
┌────────────────▼────────────────┐
│ LLM Judge evaluation (Separate) ├──────┐
└────────────────┬────────────────┘      │
                 │                       │
      [Score >= Threshold?]              │ Feedback & Details
       ├── YA  ──► Commit to Master      │
       └── TIDAK ────────────────────────┘
```

---

## 📊 Format Kriteria Penilaian

Model Judge memberikan skor numerik dan justifikasi berdasarkan bobot berikut:

| Kriteria Penilaian | Bobot | Target Minimal | Tindakan jika Gagal |
| :--- | :--- | :--- | :--- |
| **Logic Correctness** | 40% | $9.0 / 10$ | Kirim kembali ke implementer agent untuk diperbaiki. |
| **Security Auditing** | 30% | $10.0 / 10$ | Blokir kode dan jalankan static analyzer ulang. |
| **Clarity & Indentation** | 30% | $8.0 / 10$ | Jalankan auto-formatter lokal. |

## 📝 Prompt Judge Evaluator

Berikut adalah format prompt terstruktur yang digunakan oleh Judge Evaluator untuk menganalisis kode yang dihasilkan agen:

```markdown
### SYSTEM PROMPT: CODE QUALITY ASSURANCE JUDGE

Tugas Anda adalah bertindak sebagai Security and Logic Auditor yang ketat. Nilai kode yang diberikan dengan format JSON yang memuat skor dan penjelasan.

**Kriteria Penilaian:**
1. Logic: Periksa adanya runtime error, infinite loop, atau edge cases yang tidak ditangani.
2. Security: Evaluasi kebocoran kredensial atau command injection.

**Format Output JSON:**
{
  "logic_score": 9.5,
  "security_score": 10.0,
  "justification": "Detailed analysis text...",
  "approved": true
}
```

## 🛡️ Fallback Sistem Penilaian

* **Toleransi Terbatas**: Jika setelah 2 kali backtracking nilai Judge tetap di bawah threshold, serahkan keputusan langsung ke pengguna (HITL Negotiation).
* **Linter Override**: Linter otomatis (seperti ESLint atau PyLint) memiliki wewenang override untuk mengabaikan keluhan gaya penulisan dari Judge jika aturan lokal linter sudah terpenuhi.

---

## ⚠️ Anti-Patterns Evaluasi LLM-as-a-Judge

* ❌ **Same Instance Bias**: Menggunakan instansi chat model yang sama untuk memvalidasi kodenya sendiri, yang sering menghasilkan bias pembenaran (confirmatory bias).
* ❌ **Ignoring Static Analysis**: Mengandalkan audit model LLM sepenuhnya tanpa menjalankan linter statis lokal (seperti Ruff atau ESLint).
* ❌ **No Strict Output Format**: Membiarkan model Judge memberikan jawaban dalam format narasi bebas tanpa format parsing terstruktur (JSON/YAML), yang menyulitkan otomasi regex check.
